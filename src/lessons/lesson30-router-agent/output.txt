Agent Answer:
### 一句话结论
SSE（Server-Sent Events）是服务器基于标准 HTTP 长连接，以 `text/event-stream` 格式逐行向客户端推送数据的单向流式协议，对 Java 后端而言，它本质上是**“带事件标签、自动重连和断点续传能力的 HTTP Chunked 输出”**。

---

### 原理拆解（Java 工程师视角）

| 传统 HTTP 接口 | SSE 流式输出 |
|----------------|--------------|
| `CompletableFuture.get()`：阻塞等待全量结果 | `Stream<T>` / `Publisher<T>`：按需 `flush()` 逐块下发 |
| 一次请求一次响应，连接立即关闭 | 保持 TCP 连接存活，服务端主动 `write`，客户端被动 `read` |
| 适合批量返回 | 适合增量渲染（如 AI 打字机效果、进度条、日志流） |

**底层机制类比：**
1. **传输层**：复用 HTTP/1.1 的 `Transfer-Encoding: chunked`，不依赖 WebSocket 的握手协议，直接走普通 GET 请求。
2. **数据格式**：服务端按固定规则写流：
   ```
   id: 1\r\n
   event: message\r\n
   data: {"text":"Hello"}\r\n
   \r\n
   ```
   客户端（浏览器 `EventSource`）按 `\r\n` 切割，遇到连续两个换行符视为一个完整事件，自动触发对应回调。
3. **连接生命周期**：类似 Servlet 异步请求（`AsyncContext`），但框架接管了心跳检测、断线重连、`Last-Event-ID` 协商等细节，你只需关注“何时发数据”和“何时关连接”。
4. **与 WebSocket 的本质区别**：WebSocket 是双向管道（像 `SocketChannel`），SSE 是单向广播（像 `Kafka Consumer` 的简化版）。SSE 天然兼容 HTTP 中间件（网关、CDN、Nginx 缓存/限流），且浏览器零依赖。

---

### 工程落地理解（生产级要点）

#### 1. Spring Boot 中的标准用法
```java
@GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter stream() {
    SseEmitter emitter = new SseEmitter(60_000L); // 60s 超时
    executor.submit(() -> {
        try {
            for (String chunk : aiResponseChunks()) {
                emitter.send(SseEmitter.event().name("chunk").data(chunk));
                Thread.sleep(50); // 模拟延迟
            }
            emitter.complete();
        } catch (Exception e) {
            emitter.completeWithError(e);
        }
    });
    return emitter;
}
```
> 💡 `SseEmitter` 底层依赖 Servlet 3.0+ 异步模型，**每个连接会占用一个 Tomcat 工作线程**，不可滥用。

#### 2. 生产环境必踩的 4 个坑
| 问题 | 根因 | 解决方案 |
|------|------|----------|
| **连接数爆炸导致 OOM/EPOLL 耗尽** | 每个 SSE 连接持有一个线程 + socket fd | 设置 `max-connections`（Redis 计数或本地 Guava Cache），超限返回 `429 Too Many Requests` |
| **Nginx 提前切断连接** | Nginx 默认 `proxy_read_timeout=60s`，无数据即断 | 配置 `proxy_read_timeout 300s;` + 服务端每 15s 发送 `:keepalive\n\n` 保活 |
| **客户端断开未清理资源** | 后台线程继续跑，DB 游标/内存泄漏 | 注册 `emitter.onCompletion(() -> cancelTask())` 和 `emitter.onTimeout(...)`，确保协程/线程池安全退出 |
| **网络抖动导致重复消费** | 客户端自动重连，服务端未做幂等 | 利用 `Last-Event-ID` 头实现断点续传，或为每条数据分配单调递增 `id`，客户端去重 |

#### 3. 架构选型决策树
- ✅ 选 SSE：服务端单向推文本/进度/日志，需兼容现有 HTTP 网关，客户端为浏览器/移动端 WebView
- ✅ 选 WebSocket：需要双向实时交互（如在线聊天、协同编辑、游戏状态同步）
- ✅ 选传统 REST + 分页/游标：一次性大响应，或客户端可接受 1~3s 轮询成本

#### 4. 现代演进提示
- Spring WebFlux 中推荐使用 `Flux<SseEvent>` + Reactor 背压控制，避免内存堆积
- 云原生场景下，SSE 连接不适合跨 Pod 路由（除非用 Gateway 会话粘滞），通常配合消息队列（Kafka/RabbitMQ）做解耦：`业务服务 → MQ → SSE 网关服务 → 客户端`

掌握 SSE 的核心不是调通 Demo，而是**把“流式输出”当作有状态的资源来管理**：限流、保活、断线恢复、优雅降级，这四件事做扎实，才能扛住高并发下的真实流量。

Agent Answer:
以下是使用原生 `fetch` 读取 SSE（Server-Sent Events）的 TypeScript 最小示例。代码已处理网络分块边界、流式解码和基础事件解析，可直接运行。

### 📦 客户端代码 (`sse-client.ts`)
```typescript
async function connectSSE(url: string) {
  const response = await fetch(url);
  
  // 1. 检查 HTTP 状态
  if (!response.ok) {
    throw new Error(`HTTP ${response.status}: ${response.statusText}`);
  }

  // 2. 验证 Content-Type
  const contentType = response.headers.get('content-type') || '';
  if (!contentType.includes('text/event-stream')) {
    throw new Error('服务端未返回 text/event-stream 类型');
  }

  // 3. 安全获取可读流
  if (!response.body) throw new Error('响应体为空');

  const decoder = new TextDecoder('utf-8');
  let buffer = '';

  try {
    // 4. 逐块读取流数据
    for await (const chunk of response.body) {
      // 追加新数据到缓冲区
      buffer += decoder.decode(chunk, { stream: true });
      
      // 按换行符分割，保留最后一个不完整的片段在 buffer 中
      const lines = buffer.split('\n');
      buffer = lines.pop() ?? '';

      // 5. 解析完整行
      for (const line of lines) {
        if (line.startsWith('event:')) {
          console.log('[Event]', line.slice(6));
        } else if (line.startsWith('data:')) {
          console.log('[Data] ', line.slice(5));
        } else if (line === '') {
          // 空行表示一个完整事件结束
          console.log('------------------');
        }
      }
    }
  } catch (error) {
    console.error('SSE 连接中断或发生错误:', error);
  } finally {
    // 确保资源释放（部分环境需要显式关闭）
    if ('releaseLock' in response.body) {
      response.body.cancel?.();
    }
  }
}

// 使用示例
connectSSE('http://localhost:3000/sse').catch(console.error);
```

---

### 🔍 关键点说明
| 关键点 | 说明 |
|--------|------|
| **为什么用 `for await...of`** | `fetch` 返回的 `response.body` 是 `ReadableStream`。现代环境（Node 18+ / 浏览器）支持异步迭代，比手动 `getReader()` 更简洁。 || **缓冲区处理** | 网络分包可能导致一条 SSE 消息被切分到多个 `chunk` 中。通过 `buffer` 累积并按 `\n` 分割，可避免解析断裂。 |
| **SSE 协议格式** | 每行以 `key: value\n` 结尾，事件之间用空行 `\n\n` 分隔。示例仅解析了 `event:` 和 `data:`，生产环境需处理多行 `data:` 和 `id:`/`retry:`。 |
| **错误与断开** | SSE 断开时 `for await` 会抛出错误或正常结束。建议在实际项目中加入指数退避重连逻辑。 |

---

### 🧪 配套测试服务器（Node.js 内置模块，无需依赖）
保存为 `server.ts` 并运行 `npx ts-node server.ts`：
```typescript
import http from 'http';

const server = http.createServer((req, res) => {
  if (req.url !== '/sse') {
    res.writeHead(404);
    res.end();
    return;
  }

  // SSE 必需的头信息
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    Connection: 'keep-alive',
    'Access-Control-Allow-Origin': '*'
  });

  let count = 0;
  const timer = setInterval(() => {
    // SSE 标准格式：event/data 后必须跟 \n，事件结束需额外 \n
    res.write(`event: tick\ndata: {"time":"${new Date().toISOString()}","count":${++count}}\n\n`);
  }, 1000);

  // 客户端断开时清理定时器
  req.on('close', () => clearInterval(timer));
});

server.listen(3000, () => console.log('✅ SSE Server running at http://localhost:3000/sse'));
```
